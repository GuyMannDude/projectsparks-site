Sparks Bus

Agent-to-agent messaging with delivery confirmation.

Multi-agent systems break down silently. Agent A sends a task to Agent B. Did Agent B get it? Did it process it? Did it reply? You end up asking, copy-pasting, manually relaying status. Sparks Bus closes that loop.

View on GitHub Install Guide
The Doctrine

Discord is the doorbell. Mnemo is the mailbox. The tracking ID is the receipt.

Discord never carries the payload. Mnemo carries the package. Discord tells the receiving agent where to look and gives the sender confirmation that the package was picked up.

You stop asking agents "did you get that?" — you watch #dispatch and see every package move.

Failure is a one-shot ⚠️ in #alerts.
No retry storms. No silent drops.

Install

Two install modes, auto-detected at startup

The watcher probes Mnemo at startup and picks the right mode. One codebase, two install paths.

Full Mode
With Mnemo Cortex. Each message's payload is saved to Mnemo by tracking ID. Discord notifications carry just the receipt. Semantic recall, cross-agent memory, the full doctrine.
Standalone Mode
Without Mnemo. Payload travels in the Discord notification body itself. Same delivery → notify → pickup → reply lifecycle. No semantic recall, but you can drop in Mnemo later without changing any agent code.
Standalone is the on-ramp.
Start standalone. Add Mnemo Cortex when you want semantic recall. Zero migration cost — the watcher just notices Mnemo is reachable on next restart.
  $ git clone https://github.com/GuyMannDude/sparks-bus.git
  $ cd sparks-bus
  $ pip install -r requirements.txt
  $ echo "$DISCORD_BOT_TOKEN" > ~/.sparks/discord-token
  $ cp config.example.json config.json   # edit agents/channels
  $ python3 sparks-bus-watcher.py

  Mode: STANDALONE (Discord only — payload in notifications)
  Polling every 30s
A2A Compatible

Sparks Bus does for agent-to-agent what MCP does for agent-to-tool.

Every agent in a Sparks Bus deployment has an A2A Agent Card in agent-cards/ describing identity, capabilities, and delivery method. Each bus message maps cleanly to an A2A Task:

  Sparks Bus       →  A2A Task
  ─────────────────────────────────────────
  tracking_id      →  task.id
  subject          →  task.name
  body             →  task.input
  lifecycle        →  task.state
                       (CREATED/DELIVERED→submitted,
                        PICKED UP→working,
                        REPLIED→completed,
                        DELIVERY FAILED→failed)
  reply_to         →  task.metadata.reply_to
  Mnemo session_id →  task.artifact

HTTPS / JSON-RPC transport is the v2 roadmap. Data shape compatibility is in now — when transport ships, external A2A clients can submit tasks to your Sparks agents without any data-model changes.

Idea File as Publishing Format

Bootstrap a deployment by pasting one prompt into your AI agent

The repo includes SETUP-PROMPT.md — a self-contained prompt any AI agent can read to build the entire bus on a fresh machine. Architecture, schema, lifecycle, A2A mapping, and step-by-step build instructions, all in one paste.

Karpathy's "idea file as publishing format" pattern. Don't ship a tool — ship a description an AI can build from.
